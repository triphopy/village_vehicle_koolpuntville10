Architecture Overview

Webhook layer

doPost(e) รับ webhook จาก LINE, parse payload, สร้าง requestId และ route event ไปยัง handler ที่เหมาะสม

Handlers

textHandler, imageHandler, followHandler แยก flow ข้อความ, OCR และ follow event ออกจากกัน

Services

รวม logic สำหรับ LINE API, OCR, vehicle search, visitors, logs, maintenance และ HTTP retry wrapper

Admin command layer

คำสั่ง admin ถูกแยกเป็นไฟล์ย่อย เช่น /health, /syslog, /testalert, /version

Runtime flow แบบย่อ

01
LINE ส่ง event มาที่ GAS Web App

doPost(e) เป็น entrypoint หลักของระบบ

02
ระบบ route event ไปยัง handler

ข้อความจะเข้า textHandler, รูปภาพจะเข้า imageHandler

03
handler เรียก services ตามประเภทงาน

เช่น lookup staff, search vehicle, OCR, logging, visitor tracking

04
admin commands ถูก dispatch แยกตามคำสั่ง

ลด conflict และทำให้แก้ command รายตัวได้ง่าย

Admin Commands

Command Use case หมายเหตุ
/add, /remove, /setstatus, /setroleจัดการข้อมูล Staffใช้งานผ่าน LINE ได้โดยตรง
/list, /status, /whois, /visitorsตรวจข้อมูลผู้ใช้และการใช้งานเหมาะกับ support และ auditing เบื้องต้น
/log <count>อ่าน search logs ล่าสุดflush buffered logs ก่อนอ่าน
/syslog <count>อ่าน system logs ล่าสุดแสดงเวลา, level, source, event, requestId
/healthตรวจสุขภาพระบบแบบ defaultไม่ยิง LINE/Gemini live check
/health fullตรวจสุขภาพระบบแบบ liveยิง LINE Profile API และ Gemini API จริง
/testalertยืนยันว่า LINE admin alert และ SystemLog ยังทำงานได้ใช้เทส staging/prod โดยไม่ต้องทำให้ระบบ fail
/clearcacheล้าง runtime cacheใช้หลังแก้ชีตแล้วผลยังไม่อัปเดต
/versionดู deployment version และ GitHub SHAอาศัย `GITHUB_REPO` สำหรับ latest commit check

Ops, Logging และ Maintenance

HTTP layer และ retry

httpService.js รวม logic ของ fetchWithRetry() และ fetchJsonWithRetry() เพื่อใช้กับ LINE API, Gemini API และ GitHub API โดยให้ retry และจัดหมวดหมู่ error กลางที่เดียว

Buffered logging

  • Log buffer สูงสุด 10 รายการก่อน flush
  • SystemLog buffer สูงสุด 5 รายการสำหรับ WARN และ INFO
  • ALERT และ ERROR flush ลงชีตทันที

Health check behavior

  • /health ตรวจ config, spreadsheet, cache, drive และสร้าง SystemLog ให้ถ้ายังไม่มี
  • /health full เพิ่ม live checks ของ LINE และ Gemini
  • health summary จะถูกเขียนลง SystemLog เมื่อมี warn, fail หรือ response ช้า

Maintenance jobs

Function หน้าที่
keepAlive()ใช้เป็น trigger เบา ๆ เพื่อให้ script ตื่นอยู่เมื่อจำเป็น
dailyBackup()สำรอง spreadsheet ไปยังโฟลเดอร์ backup
cleanOldBackups()ลบ backup เก่าตาม retention
dailyCleanup()ล้างข้อมูลเก่าจาก `Log`, `Visitors`, `SystemLog`
dailyMaintenance()run backup + cleanup + alert เมื่อเกิด partial failures
เมื่อ maintenance step ใด fail ระบบจะส่ง LINE alert ไปยัง admin และเขียน maintenance_partial_failure ลง SystemLog

Deployment Flow

01
Push code ไปยัง GitHub

branch main ใช้สำหรับ production และ feature/** ใช้สำหรับ staging

02
GitHub Actions รัน workflow

workflow จะติดตั้ง @google/clasp และเขียนไฟล์ credentials ที่จำเป็น

03
CI เขียน src/versionInfo.js

เพื่อ inject branch, SHA และ metadata ของ deployment ล่าสุด

04
Deploy ด้วย clasp

รัน clasp push --force แล้วตามด้วย clasp deploy

main        -> BLUE / production GAS
feature/**  -> GREEN / staging GAS
